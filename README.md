# HotelWebApp (Сотрудники сети отелей)

Минималистичная платформа для сотрудников отелей: оформление/отмена брони, добавление гостей, просмотр брони по номеру и ролевая модель с ограничением по корпусу.

## Стек
- Frontend: React (Vite + TypeScript)
- Backend: Node.js (Express + TypeScript)
- Database: PostgreSQL

## Быстрый старт (локально)
1. Установите Node.js >= 18 и PostgreSQL (если не используете Docker).
2. Поднимите БД:
   - `docker compose up -d`
3. Настройте переменные окружения:
   - скопируйте `.env.example` в `.env` (в корне проекта)
4. Первый запуск API:
   - `cd server && npm i && npm run dev`
   - при первом старте сервер выполнит SQL-миграции из `db/migrations` (если включено `DB_AUTO_MIGRATE=true`)
5. Запуск фронтенда:
   - `cd client && npm i && npm run dev`

Альтернатива: запуск API командой `npm run dev` из корня проекта (обёртка в `HotelWebApp/package.json`).

## Быстрый старт (через Docker, без Node на хосте)
1. Убедитесь, что Docker запущен.
2. Выполните в корне проекта:
   - `docker compose up --build`
3. Откройте:
   - Frontend: `http://localhost:5173`
   - API health: `http://localhost:4000/api/health`

## Администратор для создания учетных записей
После первого запуска API при отсутствии аккаунтов создается bootstrap `network_admin`:
- `BOOTSTRAP_ADMIN_EMAIL`
- `BOOTSTRAP_ADMIN_PASSWORD`

Значения по умолчанию заданы в `.env.example`.

## Важные параметры безопасности
- брутфорс на логин (rate limiting)
- строгая валидация входных данных (Zod)
- RBAC по должности и ограничение по корпусу для сотрудников
- параметризованные SQL-запросы (pg)

## Как менять данные в базе без фронтенда (локально без Docker)

Сервер при старте **автоматически применяет SQL-миграции** из каталога `db/migrations` (упорядоченно по числовому префиксу `001_*.sql`, `002_*.sql`, …), если включено переменное окружение **`DB_AUTO_MIGRATE=true`** (такое значение стоит по умолчанию в логике сервера, если переменную не переопределять).

Чтобы изменения точно попали в БД после перезапуска API:

1. Добавьте **новый** файл миграции с **следующим номером**, например `005_my_changes.sql`.
2. Перезапустите API (`npm run dev` из корня проекта или `npm run dev` из `server`) — транзакция миграции выполнится один раз и запишется строка в таблицу `schema_migrations`.

Важные ограничения:

- **Правки уже применённого файла** (например, вы поменяли `002_*`, а в `schema_migrations` уже есть `id=2`) **не переигрываются** автоматически. Для «пересборки с нуля» проще всего пересоздать базу/схему либо вручную удалить соответствующую строку из `schema_migrations` и понимать риски (обычно безопаснее добавить новую миграцию).
- Массовый демонстрационный сид живёт отдельно в `004_bulk_demo_seed.sql` и помечает одноразовый прогон в таблице `demo_seed_runs`, чтобы не дублировать сотни броней при каждом старте при повторном применении этого же файла.

Ручная альтернатива (не обязательна): выполнить SQL через `psql`, но тогда **ID миграции придётся отметить в `schema_migrations` самостоятельно**, иначе сервер попытается применить тот же файл повторно.
